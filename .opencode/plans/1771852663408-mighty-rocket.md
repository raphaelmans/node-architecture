# Plan: Apply TDD Workflow to Core Testing Standards

## Goal

Make TDD (red -> green -> refactor, vertical slicing, behavior-first tests) explicit in the two canonical testing standards the user named:

- `client/core/testing.md`
- `server/core/testing-service-layer.md`

## Scope

Only update the two files above. No new docs, no restructuring, no cleanup outside this request.

Current constraint from user:

- Tests are stubbed-boundary for now and run offline/fast.
- No live server or live database interaction.
- Behavior remains primary: tests describe externally observable outcomes.
- Tests go through public interfaces/service functions, not private methods.
- Avoid internal call-count/order assertions unless ordering is the behavior under test.
- External services (e.g., Supabase) are stubbed at the boundary using test doubles (Vitest mocks, MSW, or equivalent).

Canonical wording to encode in both docs:

"TDD Behavioral Testing with Stubbed Boundaries: We follow the Red-Green-Refactor cycle using vertical slices — one failing test, one implementation, repeat. Tests are behavior-focused, describing what the system should do rather than how it does it internally. Tests go through public interfaces and service functions, never testing private methods or asserting on internal call counts. External services are stubbed at the boundary using test doubles, so tests run fast, offline, and without a live database."

Skill reference requirement from user:

- If skill support exists in the agent environment, explicitly use/load `tdd`.
- In both docs, add a small reference to canonical skill source:
  - `https://github.com/mattpocock/skills/tree/main/tdd`
- Keep this reference optional/guidance-level, not a hard dependency for contributors.
