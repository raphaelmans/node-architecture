Core Principles

🔍

Explicit over Implicit

No IoC magic. Dependencies are wired manually in factories — every object creation is traceable.

🧱

Strict Layer Boundaries

Each layer has one responsibility. Dependencies only flow downward. No layer skips its neighbor.

🧪

Testability First

Interface-based layers + manual DI means every layer is testable in isolation with simple mocks.

📜

Zod as Source of Truth

Schemas define contracts. Types are inferred — no duplication between runtime validation and TypeScript.

🚫

Framework-Agnostic Core

Services and use cases are pure TypeScript. They don't know about Next.js, tRPC, or any framework.

🔒

Public-Safe Errors

5xx errors never reveal internals. Stack traces and SQL are logged server-side only.

Golden Rules

1
One directionDependencies flow downward only. Router → Service → Repo. Never up.
2
Routers call one thingA router procedure calls exactly one use case or one service — never both.
3
Services don't call servicesIf two services must coordinate, create a Use Case.
4
Repos return entitiesRepositories return raw domain entities, never shaped DTOs.
5
Transactions via contextRequestContext carries the tx — services opt-in to participate via ctx.tx.
6
Kernel stays pureshared/kernel/ imports only Zod. No framework, no infra.

🌐 Layer 1

Router / Controller

The entry point for all HTTP/tRPC traffic. Parses and validates input with Zod, calls exactly one use case or service, then shapes the response. It knows about HTTP — nothing below it does.

✅ Owns

  • Input parsing & Zod validation
  • Calling exactly one use case or service
  • Mapping errors to HTTP responses
  • Omitting sensitive fields from responses

❌ Does NOT own

  • Business logic of any kind
  • Database or ORM access
  • Service-to-service calls
  • Transaction management
src/lib/modules/user/user.router.ts 
export const userRouter = router({
  register: publicProcedure
    .input(RegisterUserInput)  // ← Zod validates here
    .mutation(async ({ input, ctx }) => {
      // Call EXACTLY one use case
      const user = await makeRegisterUserUseCase()
        .execute(input);

      // Shape response — omit sensitive fields
      return { id: user.id, email: user.email };
    }),
});
🎯 Layer 2

Use Case

Orchestrates multi-service workflows. When a single user action requires multiple services to coordinate — like registration (create user + add to workspace + send email) — that belongs here. Not every operation needs a Use Case; simple single-service ops skip this layer.

✅ Owns

  • Multi-service orchestration
  • Transaction boundaries (multi-service ops)
  • Triggering side effects (emails, audit logs)
  • Complex workflow logic

❌ Does NOT own

  • HTTP / tRPC details
  • ORM-specific code
  • Business rules of individual services
  • Simple single-service operations
src/lib/modules/user/use-cases/register-user.use-case.ts 
export class RegisterUserUseCase {
  async execute(input: RegisterUserDTO) {
    // Own the tx across multiple services
    const user = await this.txManager.run(async (tx) => {
      const u = await this.userService.create(input, { tx });
      await this.workspaceService.addMember(wsId, u.id, { tx });
      return u;
    });

    // Side effects OUTSIDE the transaction
    await this.emailService.sendWelcome(user.email);
    return user;
  }
}
⚙️ Layer 3

Service

The home of business logic. A service handles a single domain concept — all rules, validations, and transformations for "User" live here. It can participate in transactions started by Use Cases, or own them for simple operations.

✅ Owns

  • Business rules & validation
  • Entity transformations
  • Business events (logging)
  • Transaction participation or ownership

❌ Does NOT own

  • Calls to other services (Use Case does this)
  • HTTP details
  • ORM specifics (Repository does this)
  • Transaction creation for multi-service ops
src/lib/modules/user/services/user.service.ts 
async create(data: UserInsert, ctx?: RequestContext) {
  // Business rule: email must be unique
  const existing = await this.repo.findByEmail(data.email, ctx);
  if (existing) throw new UserEmailConflictError(data.email);

  const run = (tx) =>
    this.repo.create({ ...data, id: newId() }, { tx });

  // Participate in external tx if provided, else own it
  if (ctx?.tx) return run(ctx.tx);
  return this.txManager.run((tx) => run(tx));
}
🗄️ Layer 4

Repository

The only layer that talks to the database. Repositories abstract away the ORM (Drizzle) so business logic never contains SQL. They accept an optional transaction context and always return plain entities.

✅ Owns

  • SQL / ORM queries
  • Entity ↔ database row mapping
  • Transaction participation via context
  • Null handling (return null or throw)

❌ Does NOT own

  • Business logic
  • Transaction creation
  • Logging
  • Returning DTOs or shaped objects
src/lib/modules/user/repositories/user.repository.ts 
async findByEmail(
  email: string,
  ctx?: RequestContext
): Promise<User | null> {
  // Use tx from context if provided, else use db directly
  const client = ctx?.tx ?? this.db;

  const [row] = await client
    .select()
    .from(users)
    .where(eq(users.email, email));

  return row ?? null;  // return entity, never DTO
}
💾 Layer 5

Database

PostgreSQL accessed via Drizzle ORM. Drizzle provides full type-safety with zero runtime overhead — your schema is the source of truth for both DB and TypeScript types. The connection is serverless-optimized via postgres.js.

✅ Owns

  • Data persistence
  • ACID transactions
  • Indexing & query optimization
  • Schema migrations

❌ Does NOT own

  • Business logic (none should reach here)
  • Application-level validation
src/lib/shared/infra/db/schema.ts 
// Drizzle schema = TypeScript types + DB schema
export const users = pgTable("users", {
  id:        text("id").primaryKey(),
  email:     text("email").notNull().unique(),
  name:      text("name").notNull(),
  createdAt: timestamp("created_at").defaultNow(),
});

// Type inferred from schema automatically
type User = typeof users.$inferSelect;

Route / Page

✅ Owns

❌ Does NOT own

  • Feature business logic
  • Data fetching rules
  • Cache management
  • Form handling

🧩 Layer 2

Feature Component

The business orchestrator on the frontend. It wires together data-fetching hooks, mutation handlers, and presentation components. It manages loading states, errors, and user flows — but delegates rendering to dumb components.

✅ Owns

  • Calling query adapter hooks
  • Wiring loading / error states
  • Form orchestration
  • User flow logic (multi-step, conditional)

❌ Does NOT own

  • HTTP / tRPC transport
  • Cache key management
  • Direct API calls
  • Pure rendering (leave to Presentation)
src/features/user/components/UserProfileFeature.tsx 
export function UserProfileFeature({ userId }) {
  const { data, isLoading, error } = useQueryUserProfile(userId);
  const updateMut = useMutUpdateUser();

  if (isLoading) return <Spinner />;
  if (error) return <ErrorAlert error={error} />;

  return (
    <UserProfileCard
      user={data}
      onUpdate={(v) => updateMut.mutate(v)}
    />
  );
}
🪝 Layer 3

Query Adapter (Hooks)

Bridges the Feature API and TanStack Query. It owns cache keys, invalidation logic, and optimistic updates. Components never call the API directly — they always go through these hooks.

✅ Owns

  • Query key definitions
  • Cache invalidation strategy
  • Optimistic updates
  • Infinite scroll / pagination

❌ Does NOT own

  • HTTP transport
  • Error normalization
  • Endpoint path strings
  • UI state
src/features/user/hooks.ts 
export function useQueryUserProfile(userId: string) {
  const api = getUserApi();
  return useQuery({
    queryKey: ['user', 'profile', userId],
    queryFn: () => api.getProfile(userId),
  });
}

export function useMutUpdateUser() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (data) => getUserApi().update(data),
    onSuccess: () => qc.invalidateQueries({ queryKey: ['user'] }),
  });
}
🔌 Layer 4

Feature API

An interface + class that owns all API calls for a single feature. It normalizes errors from any transport into the AppError discriminated union. Think of it as the adapter between transport and your app.

✅ Owns

  • Endpoint paths / tRPC procedures
  • DTO → Feature model mapping
  • Error normalization (unknown → AppError)
  • Zod response schema validation

❌ Does NOT own

  • Cache management
  • HTTP client config
  • React state
  • Business rules
src/features/user/api.ts 
interface IUserApi {
  getProfile(id: string): Promise<UserProfile>;
}

class UserApi implements IUserApi {
  async getProfile(id: string) {
    try {
      const dto = await this.deps.clientApi.user
        .getProfile.query({ id });
      return parseUserProfile(dto); // Zod parse
    } catch (err) {
      throw this.deps.toAppError(err); // normalize
    }
  }
}
📡 Layer 5

Client API

The lowest frontend layer — a thin wrapper around the HTTP client (tRPC, ky, axios). It handles base URL, auth headers, and the standard response envelope. Feature APIs call this; nothing else does.

✅ Owns

  • HTTP client configuration
  • Auth headers / session tokens
  • Response envelope unwrapping
  • Global request/response interceptors

❌ Does NOT own

  • Endpoint paths
  • Domain logic
  • Cache management
  • Error normalization

🎨 Layer 6

Presentation Components

Pure render-only components. They receive all data via props and emit events via callbacks. No hooks, no fetching, no mutation calls. The easiest components to test and reuse across different contexts.

✅ Owns

  • HTML structure and styling
  • Conditional rendering based on props
  • User interaction callbacks
  • Accessibility attributes

❌ Does NOT own

  • Any data fetching
  • Cache or mutation logic
  • Navigation
  • Business rules

What's happening

Click Animate to trace a registration request, or click any node to learn about it.
Browser
The user fills in the registration form and clicks Submit. React hook-form validates input locally via Zod, then calls the mutation hook: useMutRegisterUser().mutate({email, name})
Feature API
The mutation hook calls UserApi.register(). The Feature API sends the tRPC request and wraps any error into a typed AppError. Transport details never escape this layer.
tRPC Router
The request arrives server-side. The publicProcedure validates input with Zod. If valid, it calls makeRegisterUserUseCase().execute(input) — exactly one thing.
Use Case
RegisterUserUseCase opens a database transaction and calls multiple services inside it atomically. If anything fails, the whole transaction rolls back cleanly.
Service
UserService.create() checks if email is taken, generates an ID, then calls the repository. It receives the transaction context (ctx.tx) and participates in it automatically.
Repository
The only code that speaks SQL. It uses ctx?.tx ?? this.db to write within the active transaction. Returns a plain entity — never a DTO.
PostgreSQL
The transaction commits atomically — both the user row and workspace membership are either fully written or fully rolled back. The response flows back up the stack.

Design Patterns

Click any pattern to expand the explanation and code example.

🏗️
Layered Architecture
Structural
▼

The entire system is organized into layers, each with a single responsibility. Dependencies flow in one direction only: downward. A Router knows about Use Cases; a Use Case doesn't know about HTTP.

Router   → UseCase / Service (HTTP entry point)
UseCase  → Service (orchestration)
Service  → Repository (business rules)
Repo     → Database (data access)

// ❌ This BREAKS the pattern:
class UserService {
  doSomething() {
    this.trpcRouter.callOtherRoute(); // ❌ NO!
  }
}
💉
Manual Dependency Injection
DI Pattern
▼

No IoC container magic. Dependencies are created in factory functions and passed explicitly. This makes the dependency graph visible and easy to replace in tests.

// ✅ Explicit wiring in factory
function makeUserService() {
  return new UserService(
    makeUserRepository(), // explicit!
    getContainer().txManager,
  );
}

// ✅ Tests swap with a fake easily
const svc = new UserService(
  new FakeUserRepository(),
  new FakeTxManager(),
);
📦
Repository Pattern
Data Access
▼

Repositories are the only code allowed to speak SQL or Drizzle. Business logic never knows what database engine is used — it works against an interface.

// Interface — business layer sees this
interface IUserRepository {
  findById(id: string, ctx?: RequestContext): Promise<User | null>;
}

// Drizzle implementation — invisible to business logic
class UserRepository implements IUserRepository {
  async findById(id, ctx?) {
    const db = ctx?.tx ?? this.db;
    const [r] = await db.select().from(users).where(...);
    return r ?? null;
  }
}
🏭
Factory Pattern
Creational
▼

Factories own the creation of objects. Each module has a factory file that lazily creates singletons for stateless objects (services, repos) and fresh instances for stateful ones (use cases).

// Lazy singleton — safe for serverless warm starts
let _svc: UserService;
export function makeUserService() {
  return (_svc ??= new UserService(...));
}

// Per-invocation for stateful use cases
export function makeRegisterUserUseCase() {
  return new RegisterUserUseCase(
    makeUserService(),
    makeEmailService(),
    getContainer().txManager,
  );
}
🔄
Context Propagation
Transaction
▼

RequestContext carries the database transaction and requestId through the call stack. Services and repositories opt-in by checking ctx?.tx.

type RequestContext = {
  tx?:       Transaction;  // database tx (optional)
  requestId?: string;      // for log correlation
};

// Pattern in every Service + Repository:
async create(data, ctx?: RequestContext) {
  const client = ctx?.tx ?? this.db;
  // ↑ use existing tx, or fallback to direct db
}
🛡️
Error Hierarchy
Error Handling
▼

All errors extend AppError. Domain errors extend HTTP error classes. This gives type-safe catching, unique error codes, and public-safe messages without manual mapping.

// Base → HTTP → Domain
abstract class AppError extends Error { ... }

class ConflictError extends AppError {
  httpStatus = 409;
  code = "CONFLICT";
}

class UserEmailConflictError extends ConflictError {
  code = "USER_EMAIL_CONFLICT";
  constructor(email: string) {
    super("Email already in use", { email });
  }
}
📋
Zod-First Contracts
Validation
▼

Zod schemas are the single source of truth. Types are inferred from schemas — never written twice. The same schema validates at the API boundary and in the Feature API.

// Schema = runtime validation + TypeScript type
const RegisterInput = z.object({
  email: z.string().email(),
  name:  z.string().min(2),
});

// Type INFERRED — never written manually
type RegisterInput = z.infer<typeof RegisterInput>;
// → { email: string; name: string }
🔔
Envelope Pattern
API Design
▼

Every API response follows the same shape: success returns { data: T }, errors return { code, message, requestId }. Client handling is consistent and predictable.

// Success
{ data: { id: "usr-123", email: "a@b.com" } }

// Paginated
{ data: [...], meta: { total, cursor, nextCursor } }

// Error (4xx or 5xx)
{
  code: "USER_EMAIL_CONFLICT",
  message: "Email already in use",
  requestId: "req-abc-123"
}

Frontend AppError Union

export type AppError =
  | { kind: "network";      message: string }
  | { kind: "unauthorized"; message: string; ... }
  | { kind: "forbidden";    message: string; ... }
  | { kind: "not_found";    message: string; ... }
  | { kind: "validation";   fieldErrors?: ...; ... }
  | { kind: "rate_limited"; message: string; ... }
  | { kind: "unknown";      message: string };

// UI branches ONLY on kind — transport never leaks
if (error.kind === "validation") {
  // show field-level errors
} else if (error.kind === "rate_limited") {
  // show retry UI
}

How Factories Wire Dependencies

src/lib/modules/user/factories/user.factory.ts 
import { UserRepository } from "../repositories/user.repository";
import { UserService } from "../services/user.service";
import { getContainer } from "../../shared/infra/container";

// Lazy singletons — created once, reused on warm starts (serverless-friendly)
let userRepo: UserRepository;
let userService: UserService;

export function makeUserRepository() {
  if (!userRepo) userRepo = new UserRepository(getContainer().db);
  return userRepo;
}

export function makeUserService() {
  if (!userService) {
    userService = new UserService(
      makeUserRepository(),
      getContainer().transactionManager,
    );
  }
  return userService;
}

// Use cases are per-invocation (they can carry per-request state)
export function makeRegisterUserUseCase() {
  return new RegisterUserUseCase(
    makeUserService(),
    makeWorkspaceService(),
    makeEmailService(),
    getContainer().transactionManager,
  );
}
